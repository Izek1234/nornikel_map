# План дальнейших работ

Дата анализа: 3 июля 2026. Дедлайн из ТЗ: 4 июля, 23:59.

## 1. Текущее состояние

Каркас покрывает основную идею решения:

- FastAPI backend, Next.js frontend и Neo4j как целевое хранилище.
- Парсинг PDF, DOCX, TXT, MD и CSV, разбиение текста на чанки.
- Онтология из 8 типов сущностей и набора допустимых связей.
- LLM-извлечение сущностей, связей и параметров экспериментов.
- Экраны документов, GraphRAG-чата и интерактивного графа.
- Демо-режим со встроенными сущностями, фактами и источниками.
- Нормализация части единиц, числовых диапазонов и RU/EN-синонимов.

При этом сейчас это демонстрационный прототип, а не сквозной production-MVP.

### Критические разрывы

1. Production-ветка backend неработоспособна: `main.py` вызывает отсутствующие в `graph_db.py` функции `get_stats_detailed`, `list_experiments`, `create_experiment`, `search_context`, `get_experiment_card`, а в `postprocess.py` отсутствует `build_context`.
2. Вызовы `upsert_entity` не соответствуют её сигнатуре. Проверка обычных связей всегда получает типы `Entity -> Entity`, поэтому отбрасывает все типизированные связи.
3. Активный `llm_client.py` не возвращает обычные факты, а pipeline их не сохраняет. Числа, география, confidence и цитаты фактически остаются только в demo-данных.
4. Было два расходящихся LLM-контура: устаревший legacy-слой и новый YandexGPT/Ollama (`llm_client.py`, `main.py`). README и frontend раньше описывали старый контур.
5. API-контракт frontend не совпадает с backend: health, stats и chat имеют разные поля. В production чат не возвращает `facts` и `cached`.
6. Пакетный импорт был устаревшим: неверные пути к `backend`/`data`, старая привязка к прежнему LLM-провайдеру, заявленная поддержка PPTX без парсера.
7. Нет тестов. Python компилируется и demo API проходит smoke-проверку, но TypeScript не проходит из-за отсутствия `@types/d3-force`; lint не настроен. Production build зависит от загрузки Google Fonts.
8. Корпус данных из кейса локально не добавлен, Neo4j и LLM credentials не настроены.

## 2. Критический план до сдачи

### P0.1. Восстановить сквозной pipeline

- Оставить один LLM-клиент с YandexGPT как основным провайдером и Ollama как локальным fallback.
- Сделать единый формат extraction: `entities`, `relations`, `facts`, `experiments`.
- Вынести обработку одного документа в общий сервис для API и пакетного импорта.
- Реализовать недостающие Neo4j-функции и привести сигнатуры к одному контракту.
- Сохранять сущности, связи, факты, цитаты, документ, чанк, географию и confidence.
- Использовать канонический ключ и aliases для дедупликации RU/EN терминов.
- При ошибке обработки отмечать документ как `failed`, сохраняя понятную причину.

Результат: загруженный документ появляется в Neo4j, графе и ответах чата.

### P0.2. Подключить реальные данные

- Скачать и осмотреть корпус кейса, выбрать 10-20 наиболее показательных документов.
- Настроить Neo4j и LLM через `.env.example`, не добавляя секреты в git.
- Исправить пакетный импорт и прогнать выбранный корпус.
- Вручную проверить извлечение чисел и единиц на 15-20 контрольных фрагментах.
- Подготовить небольшой стабильный seed-набор для демо на случай недоступности внешнего API.

Результат: демо работает на данных кейса, а не только на вручную заданных объектах.

### P0.3. Довести поиск и ответы

- Использовать единый GraphRAG: full-text seeds, обход графа на 2 уровня, релевантные чанки и факты.
- Добавить фильтры минимум по типу сущности, географии, году и диапазону числового параметра.
- Возвращать структурированный ответ: краткий вывод, варианты/методы, числа, confidence, источники и цитаты.
- Явно показывать конфликтующие факты и отсутствие данных, не генерировать ответ без опоры на контекст.
- Вернуть совместимый API-контракт для demo и production.

Результат: система уверенно отвечает хотя бы на 3 заранее выбранных вопроса из ТЗ.

### P0.4. Подготовить интерфейс к демонстрации

- Убрать из UI устаревшие упоминания старого LLM-контура и показывать реальный LLM provider/mode.
- Починить типы frontend и сделать сборку независимой от Google Fonts.
- В чате оформить источники, цитаты, confidence и географию отдельными читаемыми блоками.
- В графе показывать тип связи и удобный переход к карточке сущности.
- Добавить предустановленные запросы под главный демонстрационный сценарий.
- Понятно обозначить demo-режим и недоступность загрузки, если Neo4j не подключён.

Результат: пользователь понимает, откуда взят ответ и как перейти к исходному знанию.

### P0.5. Проверить и сдать

- Unit-тесты: числа/диапазоны, единицы, синонимы, онтология, chunking.
- Integration smoke: upload -> processing -> graph -> search -> chat.
- Frontend: typecheck, lint, production build и ручная проверка desktop/mobile.
- Зафиксировать 3 демо-сценария и ожидаемые ответы.
- Обновить README и архитектурную схему по фактической реализации.
- Развернуть решение, записать видео до 5 минут, обновить презентацию и собрать архив.

Критерий готовности: новый пользователь открывает ссылку, выполняет три сценария без ручного вмешательства и видит проверяемые источники.

## 3. После хакатонного MVP

### P1. Функциональная полнота

- Версионирование фактов, ручная экспертная корректировка и история изменений.
- Связи экспертов и лабораторий с темами, рекомендации экспертов.
- Выявление пробелов и отдельный экран сравнительного анализа технологий.
- Экспорт Markdown, PDF и JSON-LD.
- RBAC, разграничение внутренних/внешних документов и аудит действий.

### P2. Надёжность и масштаб

- Очередь фоновых задач вместо FastAPI `BackgroundTasks`, retry и идемпотентность.
- Hybrid retrieval: full-text + embeddings + графовые фильтры.
- Автоматическая оценка качества extraction и ответов на размеченном наборе.
- Наблюдаемость: метрики, трассировка, стоимость LLM, журнал ошибок.
- Нагрузочное тестирование до 1 млн сущностей и оптимизация под SLA 3-5 секунд.
- SHACL/OWL-валидация онтологии и подключение новых технологических доменов.

## 4. Рекомендуемый порядок выполнения

1. Backend pipeline и единый контракт.
2. Реальные данные и Neo4j.
3. Три качественных GraphRAG-сценария.
4. Интерфейс и прозрачные источники.
5. Автотесты, deploy, видео и презентация.

До дедлайна не стоит распыляться на полноценный RBAC, уведомления и миллион сущностей. Для оценки сильнее работает узкий, воспроизводимый сквозной сценарий с реальными данными, числами, графом и доказуемыми источниками.
